MES/MYM Systematic Signal Research — Journey Summary
======================================================
Written: 2026-03-15  |  Updated: 2026-03-31


STARTING POINT
--------------
Goal: identify a repeatable, quantifiable signal on MES (Micro E-mini S&P 500),
MYM (Micro Dow), and M2K (Micro E-mini Russell 2000) futures that predicts short-term
price continuation, suitable for automated execution on a Topstep practice account.

Data source: TopstepX REST API (api.topstepx.com), via a custom Python client
(src/topstep_client.py). All analysis used 5-minute bars fetched live.


SIGNAL HYPOTHESIS — 3σ CONTINUATION
-------------------------------------
The core idea: when a 5-minute bar moves sharply relative to recent volatility (≥ 3
standard deviations), price tends to continue in that direction rather than revert.

Standard deviation (σ) was computed as the rolling close-return std dev over the 20
bars immediately preceding and including the signal bar ("trailing σ"). Volume was
added as a secondary filter: signal bar volume must be ≥ 1.5× the trailing 20-bar
mean volume.

The signal is directional: a large up-bar predicts continuation up; a large down-bar
predicts continuation down. Targets and stops were set in multiples of σ (target = 3σ,
stop = 2σ from entry).

This was backtested across MES 1-minute bars (converted to 5-minute) spanning several
years, producing the baseline metrics: trade count, win rate, expectancy, and Sharpe.


VOLATILITY MEASURE SELECTION
------------------------------
Two approaches were compared for computing the signal bar's "size" relative to recent
volatility:

1. Close-return standard deviation — standard approach, uses only closing prices.
2. Garman-Klass (GK) volatility — uses OHLC, theoretically more efficient.

A key methodological question arose: should the signal bar itself be included in the
volatility window? It was discovered that the signal bar IS included in the trailing σ
window as implemented. When corrected to use only prior bars, the threshold effectively
drops — the move appears larger — and the number of triggers roughly doubles, but
expectancy collapses. The self-inclusion acts as an inadvertent quality filter: because
the signal bar's large return inflates σ, only moves that are genuinely extreme (~3.5σ
in true terms) pass a 3σ threshold. The decision was made to keep the self-inclusion.

Between GK and close-return σ: close-return consistently outperformed across all
window sizes. GK was retained only for display purposes in the live monitor (annualised
regime volatility, not used in signal logic).


MOMENTUM FILTER — CSR
----------------------
CSR (Cumulative Scaled Return) was added to filter out signals that occur against an
established counter-trend. It measures the cumulative momentum in the 40 minutes
(8 × 5-min bars) prior to the signal bar:

  CSR = Σ (log_return_i / σ) for i in prior 8 bars, direction-adjusted

Only signals where CSR ≥ 1.5 (i.e., at least modest prior momentum in the signal
direction) are accepted.

The critical design question: should the current bar be included in CSR? Testing showed
that including the signal bar's own large return floods ~90% of all signals through
(the signal bar dominates the sum), destroying the filter's discriminating power. CSR
uses only the 8 bars prior to the signal bar.

Volume-weighted CSR was also tested. The correlation between plain and volume-weighted
CSR was r ≈ 0.90, and plain CSR performed better on EV metrics. Volume weighting was
dropped.


TIME-OF-DAY FILTER — BLACKOUT WINDOWS
---------------------------------------
The signal was evaluated by UTC hour-of-day to identify periods with unfavorable
outcomes (P(stop) > 0.40 or low sample count).

The first 30 minutes after NYSE open (13:30–14:00 UTC / 09:30–10:00 EST) was
hypothesised as problematic based on prior experience. Analysis confirmed that the
problematic window is actually broader: 13:00–14:00 UTC (08:00–09:00 EST), which
corresponds to the peak U.S. economic data release window (NFP, CPI, Fed announcements,
etc.). This hour was added as a blackout.

The CME late-session period was also examined (initially labeled "CME close" at
20:00–21:00 UTC / 15:00–16:00 EST). Sample count was only n=41 over the full
backtest period — too thin to be reliable. This tentative blackout was removed.

DST CORRECTION (2026-03-16): The original blackout was defined as fixed UTC hours
(13:00–14:00 UTC). This correctly maps to 08:00–09:00 EST in winter but shifts to
09:00–10:00 EDT in summer — incorrectly blocking 30 minutes of market open and missing
the 08:30 EDT economic releases (which occur at 12:30 UTC in summer). All scripts were
updated to use ET local time via Python's zoneinfo module, so the blackout always covers
08:00–09:00 ET regardless of DST.

CONDITIONAL BLACKOUT (2026-03-16): A full time-of-day EV breakdown (src/backtest_tod.py)
revealed that during the 08:00–09:00 ET window, unfiltered signals are mildly negative
(EV = -0.04σ) but CSR-filtered signals are mildly positive (+0.09σ). The blackout was
therefore made conditional: it only blocks signals where CSR < 1.5. Signals that already
pass the CSR threshold are allowed through even during 08:00–09:00 ET. The 09:00–09:30
pre-open window (+0.45σ CSR-filtered) was confirmed to have no blackout and remain
tradeable.

MES final configuration: conditional blackout 08:00–09:00 ET (blocks if CSR < 1.5).

MYM-SPECIFIC BLACKOUT (2026-03-16): A MYM-specific TOD breakdown (src/backtest_tod.py
--sym MYM) revealed that MYM behaves differently from MES in several windows, consistent
with MYM (Dow) being less exposed to pre-market tech earnings than MES (S&P 500):

  08:00–09:00 ET:  MYM no-CSR = +0.1153σ, CSR-filtered = +0.0960σ  → POSITIVE for MYM
  09:00–09:30 ET:  MYM no-CSR = -0.2780σ, CSR-filtered = -0.0760σ  → negative (MES = +0.45σ)
  15:00–16:00 ET:  MYM no-CSR = -0.1590σ, CSR-filtered = -0.3748σ  → very negative (n=65)

The 08:00–09:00 blackout was removed for MYM entirely (signals are positive even without
CSR). Two unconditional blackouts were added:
  09:00–09:30 ET — pre-NYSE-open; Dow names do not benefit from tech pre-market moves
  15:00–16:00 ET — NYSE close; strong mean-reversion effect for MYM specifically

Per-instrument blackout windows are now configured in InstrumentConfig / BotInstrument:
  MES: [(8, 0, 9, 0, conditional=True)]
  MYM: [(9, 0, 9, 30, conditional=False), (15, 0, 16, 0, conditional=False)]


FINAL SIGNAL PARAMETERS
-------------------------
  Bar size:          5 minutes, clock-aligned (:00/:05/:10 — see bar alignment below)
  Trailing σ window: 20 bars (100 minutes), including signal bar
  σ threshold:       3.0 (|return / σ| ≥ 3.0)
  Volume filter:     ≥ 1.5× trailing 20-bar median volume (bars with volume ≥ 10 only;
                     requires ≥ 10 qualifying bars, else "warming up")
  CSR window:        dynamic — 4 bars (20 min) when GK vol < 8% ann;
                     8 bars (40 min) when GK vol ≥ 8% ann
  CSR threshold:     ≥ 1.5 (direction-adjusted), excluding signal bar
  Blackout:          per-instrument (DST-aware ET local time):
                       MES: 08:00–09:00 ET conditional (blocks only if CSR < 1.5)
                       MYM: 09:00–09:30 ET unconditional; 15:00–16:00 ET unconditional
  Target:            3.0 × σ_pts from entry
  Stop:              2.0 × σ_pts from entry
  Force-close:       25 minutes (5 bars) if neither bracket hit


PRODUCTION COMPONENTS
----------------------
Three scripts form the live production system:

src/topstep_client.py
  Core API client. Handles authentication, contract search, bar history, order
  placement, position management, and trade search. Order placement supports native
  API stop-loss/take-profit brackets (OCO, exchange-managed). MES quarterly contracts
  are hardcoded in MES_CONTRACTS and must be updated each quarter.

src/signal_monitor.py
  Terminal dashboard. Connects to TopstepX, streams live 5-minute bars for MES and
  MYM, and evaluates the signal in real time. Displays: current bar progress, trailing
  σ, volume ratio, CSR progress bar, regime volatility (GK, display only), and
  whether the instrument is currently in a signal state. All signal events are logged
  to logs/signals.csv for post-hoc analysis.

src/trading_bot.py
  Automated execution layer. Reads the same signal logic as signal_monitor but places
  real market orders on the Topstep practice account. Places orders with native API
  brackets so stop/target management is exchange-side. Monitors open positions and
  force-closes any position held longer than MAX_HOLD_MIN. Classifies exits as TARGET,
  STOPPED, TRAIL STOP, or TIME EXIT, and logs all trades to separate CSV files by
  signal type (logs/bot_trades.csv, logs/orb_trades.csv, logs/vwaslr_trades.csv).

  Three signal types run in parallel; only one position per instrument at a time
  (all types share the per-instrument lock):
    - CSR 3σ continuation: 2σ API bracket (safety net) + 0.5σ software trailing stop
    - ORB: 2σ bracket, LONG only, wide ORB filter
    - VWASLR: 2σ bracket (safety net) + 1.5σ software trailing stop

  Safety: hardcoded to account PRAC-V2-88916-19336808 (id=10634862). The account name
  string is verified at startup; the bot will not run against any other account.
  Run with --paper to log signals without placing orders.


DOM RECORDER
-------------
In parallel with bar data, real-time Depth of Market (DOM) data was captured via the
TopstepX SignalR WebSocket (Market Hub). The DOM provides the live order book — bid
and ask sizes at each price level — along with best bid/ask quotes and individual trade
prints.

src/dom_client.py
  Connects to rtc.topstepx.com/hubs/market and subscribes to three event streams:
  GatewayDepth (incremental order book updates), GatewayQuote (top-of-book), and
  GatewayTrades (individual prints). Maintains a live DOMBook in memory with
  thread-safe incremental updates (add/remove levels, full reset on session open).

  In --record mode, DOM snapshots are taken at regular intervals (default 5 minutes,
  aligned to clock boundaries) and appended to mes_dom_snapshots.csv. Each row is an
  ML-ready feature vector: top-10 bid/ask prices and sizes, total liquidity each side,
  bid/ask imbalance at L1/L5/L10, and largest resting order ("wall") on each side with
  its distance from mid-price.

  In display mode (default), renders a live terminal order book view refreshed twice
  per second, showing both sides of the book with size bars, spread, and total
  liquidity.

Deployed via a shell wrapper (scripts/run_dom_recorder.sh) with exponential back-off
restart logic (5s → 300s cap), which is itself managed by a macOS LaunchAgent
(com.mf-trade-bot.dom-recorder). The wrapper handles conda environment activation
since launchd does not load shell profiles.

The DOM data is being accumulated for future microstructure research — e.g., whether
order book imbalance at the time of a 3σ signal predicts continuation strength, or
whether large resting orders ("walls") near the target price reduce fill probability.


5-SECOND BAR RECORDER
-----------------------
The TopstepX API retains approximately 28 hours of 5-second bar history. To enable
future short-timeframe signal research (scaling 100-minute analysis down to 100–300
seconds), a persistent recorder was built:

src/bar_recorder.py
  Polls the API every 60 seconds, backfills all available 5-second history on first
  run, and appends new bars to CSV files (mes_hist_5sec.csv, mym_hist_5sec.csv).
  Deduplication is handled by recording the last saved timestamp and filtering all
  fetched bars accordingly.

Deployed as a macOS LaunchAgent (com.mf-trade-bot.bar-recorder) so it starts at
login, stays running, and restarts automatically after crashes. Weeks of accumulation
are required before short-timeframe signal analysis is meaningful.


BAR ALIGNMENT STUDY (2026-03-16)
---------------------------------
Tested whether the 5-minute signal edge depends on bars being clock-aligned
(:00/:05/:10) or merely on the 5-minute duration. Five offsets were tested using
src/backtest_offset.py: offset 0 closes at :00/:05/:10, offset 1 at :01/:06/:11, etc.

Results (CSR-filtered EV):
  Offset 0 (:00/:05/:10)  +0.216σ  ← standard grid   BEST
  Offset 4 (:04/:09/:14)  +0.154σ
  Offset 3 (:03/:08/:13)  +0.025σ
  Offset 2 (:02/:07/:12)  +0.002σ
  Offset 1 (:01/:06/:11)  -0.007σ  (negative)

Conclusion: the edge is strongly concentrated in clock-aligned bars. Offset 1 is
negative. The likely mechanism: most charting platforms, algos, and order flow cluster
around the standard :00/:05/:10 boundaries, creating the volume/momentum spikes that
the signal captures. Off-grid bars close mid-candle and capture noise instead.
The production system (offset 0) is optimal and correct.


TIMEFRAME STUDY (2026-03-16)
------------------------------
Timeframes 2-min, 3-min, 4-min, 5-min, 6-min, and 10-min were tested under two
parameterisation schemes:

  Option A (same candle counts — trailing=20, mom=4-8, hold=3):
    2m: -0.005σ   3m: +0.008σ   4m: +0.083σ   5m: +0.195σ   6m: -0.053σ   10m: +0.050σ

  Option B (same time windows — ~100min σ, ~40min CSR, ~15min hold):
    2m: +0.054σ   3m: -0.019σ   4m: +0.069σ   5m: +0.195σ   6m: +0.015σ   10m: -0.250σ

5-minute bars are the clear winner under both schemes. The drop-off at 6 minutes and
non-monotonic pattern with 10 minutes further supports that the clock-alignment effect
(see above) is a key component of the edge, not just bar duration.

Intra-bar signal test (src/backtest_intrabar.py): signals evaluated at each 1-minute
mark within a developing 5-minute bar were deeply negative (EV = -0.92σ to -0.46σ for
minutes 1-4). Only the closed bar has positive EV (+0.047σ). The live system includes
a bar-close guard to prevent firing on partial bars.


CROSS-INSTRUMENT CORRELATION STUDY (2026-03-16)
-------------------------------------------------
Correlation of scaled log returns across MES, MNQ (Micro Nasdaq), and MYM using full
history (7 years, ~97,500 aligned 5-min bars; MNQ data acquired from Databento):

  MES–MNQ: 0.907   MES–MYM: 0.899   MNQ–MYM: 0.758

Rolling 100-min correlation ranges (p5/p95):
  MES–MNQ: 0.731 / 0.986   MES–MYM: 0.636 / 0.990   MNQ–MYM: 0.273 / 0.968

MYM divergence from MES+MNQ consensus shows consistent next-bar reversion at z > 2.0
(EV ≈ -0.0002/bar following a positive deviation). MNQ tends to continue rather than
revert — consistent with Nasdaq leading broad market moves.

Lead/lag: essentially zero at all non-zero lags at the 5-minute level; no instrument
reliably leads another. Cross-instrument divergence trades were evaluated but rejected:
the EV margin (~0.0002/bar) is too thin relative to the bid/ask spread to trade
profitably. MNQ as a confirmation filter for MES/MYM signals was also tested but
added negligible value (-0.005σ for MES, +0.037σ for MYM with -69% signal reduction)
because the CSR filter already captures the "broad-based move" information.


DYNAMIC CSR WINDOW (2026-03-16)
---------------------------------
The fixed 8-bar (40-min) CSR window was tested for regime-dependence using
src/backtest_csr_sweep.py. CSR windows from 2 to 24 bars were swept, broken down by
GK volatility regime.

Key finding: in low-volatility regimes (GK vol < 8% annualised), a shorter 4-bar
(20-min) CSR window outperforms the 8-bar baseline for both MES and MYM. In normal and
higher-vol regimes the 8-bar window remains optimal. The 4-bar result is consistent
across both instruments and years, and is theoretically grounded: in quiet markets
momentum builds slowly and recent bars carry more signal than noise.

More granular regime windows (12-bar for Active, 20-bar for High) were found but
rejected as likely overfit given smaller samples (n=212–561 vs n=1,100+ for Low/Normal)
and non-monotonic patterns.

Final dynamic rule (same for both instruments):
  GK vol < 8%:  CSR window = 4 bars (20 min)
  GK vol ≥ 8%:  CSR window = 8 bars (40 min)

The live system reads state.gk_ann_vol (already computed for display) and applies this
lookup in evaluate() / check_signal(). The display label "Momentum(40m):" updates live
to reflect the active window.

EV improvement from dynamic CSR, two-tier rule (in-sample):
  MES: +0.196σ → +0.271σ (+0.075σ)   MYM: +0.166σ → +0.196σ (+0.030σ)

ROLLING PERCENTILE THRESHOLD (2026-03-16): The concern that the fixed 8% boundary
is arbitrary and shifts relative to ambient vol was tested via src/backtest_csr_percentile.py.
Each bar's GK vol was ranked against a trailing rolling window and the bottom-p% assigned
the 4-bar CSR window; the rest kept 8 bars.

Key results (MES):
  Absolute threshold (gk<8%→4)       n=1,011  EV=+0.2705σ  (+0.075σ vs fixed-8)
  Roll=250, p<40%→4, else→8         n=1,019  EV=+0.2655σ  (+0.070σ vs fixed-8)

Threshold stability analysis confirmed the user's concern: the 25th percentile of GK vol
ranges from 4.8% to 9.9% across years (e.g., 2024 mean=9.4%, 2021 mean=5.8%), meaning
the fixed 8% boundary sits above the median in quiet years and below the 25th percentile
in volatile years.

Decision: retain the absolute 8% threshold. The rolling percentile improves EV by only
~0.005σ over the absolute threshold (noise-level), while adding complexity (250-bar
rolling history of GK vol for percentile lookup). The absolute threshold scores highest
in-sample and is sufficient.


HOLD-PERIOD STUDY (2026-03-16)

src/backtest_hold_sweep.py swept force-close times from 1–12 bars (5–60 min)
using the full current signal config (dynamic CSR, per-instrument blackouts).
The stop/target brackets are exchange-managed OCO orders and remain active
regardless of hold period; MAX_HOLD_MIN only controls the time-exit fallback.

Results (stop=2σ, target=3σ):

  MES (n=1,071 triggers):
    3 bars (15 min): +0.267σ  ← was current
    5 bars (25 min): +0.286σ  ← new setting
    9 bars (45 min): +0.331σ  ← in-sample peak
  MYM (n=650 triggers):
    3 bars (15 min): +0.225σ  ← was current
    5 bars (25 min): +0.235σ  ← new setting
    6 bars (30 min): +0.253σ  ← local peak before non-monotonic noise

MAX_HOLD_MIN updated from 15 → 25 in both signal_monitor.py and trading_bot.py.


MNQ ORB STRATEGY EVALUATION (2026-03-20)

The momentum / 3σ continuation strategy was already known to not work for MNQ.
As a separate investigation, the Opening Range Breakout (ORB) strategy (src/backtest_orb.py)
was extended to support MNQ (mnq_hist_1min.csv, 678,804 RTH bars, 2019–2026) by adding
it to the HIST symbol map.

ORB results (LONG only, EOD hit rate / avg pts, n≈1,100–1,500 per period):
  ORB  1 min: 56.1% hit / +6.59 pts avg
  ORB  5 min: 56.1% hit / +5.23 pts avg
  ORB 10 min: 55.7% hit / +4.85 pts avg
  ORB 15 min: 56.6% hit / +5.70 pts avg
  ORB 30 min: 57.7% hit / +5.68 pts avg
  ORB 60 min: 57.3% hit / +5.14 pts avg

Key findings:
- Long signals show genuine edge across all ORB periods (55–58% EOD hit rate).
- Short signals are consistently flat to negative (~46–48% hit rate) — the asymmetry
  is persistent and would require long-only operation.
- Medium-width ORBs outperform both narrow and wide (e.g., ORB 5/10/15 min medium
  tier hits 57–59% EOD with +10–12 pts avg, vs wide ~55% / +0.5–5 pts).
- Sigma-based stop (2σ), wide ORB, morning breakout (9:45–10:30 ET) is the standout
  setup: EV = +0.44–0.52R, consistent across most years, with total backtested R of
  +49R (5-min), +117R (10-min), +147R (15-min) over 7 years.
- Tight stop at ORB high (breakout extension only) is consistently negative (EV
  = -0.12 to -0.28R); wide stop or σ-stop required.
- Decision: MES remains the focus. MNQ ORB longs show edge but the weak short side
  limits practical utility, and the research focus stays on MES.


PL-BASED CONFIDENCE SIZING (2026-03-20)
----------------------------------------
Research question: given that the 3σ signal fires 1–2 times per day, can we
increase profitability without reducing signal frequency by sizing up on the
"best" signals rather than filtering signals out?

THEORETICAL GROUNDING
The signed Path Length (PL) over the N 1-min bars preceding a signal is defined as:

  PL = Σ(log_returns) / Σ|log_returns|  ∈ [-1, +1]

+1 = price moved perfectly in one direction over N bars (maximum trending).
0  = fully choppy (as many up as down moves).
-1 = perfectly trending against the signal direction.

PL_aligned = PL × signal_direction, so positive values mean the 1-min flow
leading into the signal was trending in the same direction as the signal.

BACKTEST (src/backtest_pl_momentum.py)
Standalone test of PL + M/N (fraction of strong bars) as a predictor on 1-min
MES bars, sweeping N ∈ {10, 20, 30, 50}, MN thresh ∈ {0.50, 0.60},
PL thresh ∈ {0.50, 0.70, 0.80}, forward window K = N // 10 bars.

Best result: N=20, MN>0.60, PL>0.80 → EV = +0.39σ, but n=128 over the full
history — too sparse for a standalone signal.

FILTER TEST (src/backtest_pl_filter.py)
Tested PL_aligned and M/N as filters on top of the production 3σ signal. Attached
1-min features by fetching N 1-min bars ending at signal_ts − 5min for each trigger.

Baseline (no filter): n=1,091 triggers, EV = +0.264σ.
Best filtered combo (N=10, MN>0.50, PL>0.50): n=125 (0.06/day), EV = +0.81σ.

Problem: filtered rate drops to ~0.06 signals/day — too sparse to be practical given
only 1–2 base signals per day. Pure filtering was rejected.

QUARTILE BREAKDOWN (src/backtest_pl_sizing.py, Analysis 1)
PL_aligned quartile EVs across all 1,091 baseline triggers:
  Q1 (PL ≤ +0.14):  EV = +0.24σ  (weakest)
  Q2 (+0.14–+0.35): EV = +0.34σ
  Q3 (+0.35–+0.56): EV = +0.10σ  (worst — "grind regimes that stall")
  Q4 (PL > +0.56):  EV = +0.41σ  (best)

The non-monotonic pattern (Q3 worse than Q1) undermines simple threshold filtering
but does not affect a top-quartile sizing approach.

CONFIDENCE SIZING (src/backtest_pl_sizing.py, Analysis 2)
Keep all signals; trade 2× when PL_aligned ≥ threshold, 1× otherwise.

Results at key thresholds:
  PL ≥ +0.50 (top ~30%, n=323): +0.024σ/unit improvement, total +124R vs baseline +297R (+42%)
  PL ≥ +0.556 (top 25%, n=273): +0.027σ/unit improvement, total +111R vs baseline +297R (+37%)

The top-25% (PL ≥ +0.556) rule is positive in 7/8 years (only 2022 barely negative),
providing strong year-by-year consistency. A round threshold of PL ≥ 0.50 was chosen
for implementation: slightly more trades qualify, marginally better absolute R, and
cleaner to describe.

KAMA — INDEPENDENT REDISCOVERY OF PATH LENGTH (2026-03-20)
-----------------------------------------------------------
Perry Kaufman's "Efficiency Ratio" (1995, "Smarter Trading") is defined as:

  ER = |net price change over N bars| / Σ|each bar's change|
     = |Σ rᵢ| / Σ|rᵢ|  =  |PL|  (unsigned path length)

The signed version (PL = Σrᵢ / Σ|rᵢ|) and its use as a directional signal
strength metric and position sizing multiplier were arrived at independently.
Kaufman used ER only to drive the smoothing speed of an adaptive moving average
(KAMA); the signed directional form and confidence-sizing application are
original extensions.

Related concepts: Fractal Efficiency (same measure, different name in academic
literature), Hurst exponent (same underlying question via rescaled range).
The name "regime_pl" / "path length" used in this codebase is a cleaner
description than "efficiency ratio."

VWASLR STRATEGY (2026-03-23)
------------------------------
A third signal type — Volume-Weighted Average Scaled Log Return — was developed and
added to both signal_monitor.py and trading_bot.py.

DEFINITION
  VWASLR = Σ(volume_i × log_return_i) / (Σvolume_i × σ)
  over the last N 5-min bars. σ is computed from a slow 500-min (100 × 5-min) trailing
  window to provide a stable normalisation. The result is a signed value in σ/bar units;
  positive = volume-weighted upward drift, negative = downward drift.

SIGNAL RULE
  A LONG signal fires when VWASLR crosses above +threshold (from below); SHORT when it
  crosses below -threshold (from above). The signal represents sustained momentum in a
  direction that is confirmed by volume weighting — not a single spike bar, but a
  directional trend with volume behind it.

  RTH only: 9:30–16:00 ET (blackout windows inherited from per-instrument config).

PARAMETERS (from backtest optimisation)
  MES: N=10 bars (50 min), threshold=1.0σ/bar, EV=+0.091σ
  MYM: N=10 bars (50 min), threshold=0.5σ/bar, EV=+0.066σ

  σ-window: 100 bars = 500 min (chosen for stability; calibrated on 5-min bars only).
  Same 2σ stop / 3σ target structure as the CSR momentum signal.
  VWASLR trades logged to logs/vwaslr_trades.csv (separate from bot_trades.csv).

DISPLAY
  signal_monitor.py shows a dedicated VWASLR panel per instrument (below the main
  signal panel) with: current value vs ±threshold, a bar chart, and — when a signal
  is active — the entry price and live trailing stop level.


OVERNIGHT GLOBEX EVALUATION (2026-03-23)
-----------------------------------------
CSR and VWASLR strategies were tested during overnight Globex hours (roughly 18:00 ET
to 09:30 ET). Finding: not viable for drift-following strategies.

MECHANISM: During overnight hours, the order book is thin. What appears as sustained
directional momentum (VWASLR crossing threshold, CSR building) is partly an artefact
of low liquidity — a relatively small order causes a price spike that looks like a
signal, but the position immediately attracts adverse flow once the book refills. This
is order book fragility, not genuine price discovery.

RESULT: Overnight blackout windows were added to both instruments:
  MES: 18:00–08:00 ET unconditional (overnight Globex + pre-open)
  MYM: 18:00–09:00 ET unconditional (same, with an extended pre-open)

TWO POSSIBLE EXCEPTIONS (not yet implemented — require purpose-built approach):
  1. Sunday open: genuine price discovery after the weekend gap; real order flow.
  2. Post-NYSE-close news momentum: news-driven moves have actual order flow, not just
     drift. Different from intraday momentum in both mechanism and required response time.
  Any future Globex strategy must account for thin book depth throughout.


MFE / MAE ANALYSIS (2026-03-31)
---------------------------------
Script: src/analyze_mfe.py

For each logged trade (CSR, ORB, VWASLR), the script fetches 1-min bars over the hold
period and computes:
  MFE — Max Favourable Excursion: best price achievable from entry (peak unrealised gain)
  MAE — Max Adverse  Excursion:   worst drawdown seen from entry (peak unrealised loss)
  Capture % = actual_pnl / MFE  (how much of the peak profit was kept)

KEY FINDING: All three strategies show positive average MFE — the signal direction is
correct — but negative or near-zero actual P&L. The problem is exits, not signals.

Sample results at the time of analysis (live trades, small sample):
  VWASLR: avg MFE ≈ +42 pts  actual avg ≈ -19 pts  capture ≈ negative
  CSR:    avg MFE ≈ +65 pts  actual avg ≈ -10 pts  capture ≈ low

The gap between MFE and actual P&L motivated the trailing stop research below: if
the trade gets to its peak and then retraces all the way to the stop, the fixed stop
captures nothing. A trailing stop that locks in part of the peak gain addresses this.


TRAILING STOP BACKTEST (2026-03-31)
--------------------------------------
Script: src/backtest_trailing.py

Methodology: for each logged trade, fetches 1-min bars over the MAX_HOLD_MIN window
and simulates fixed-stop vs trailing stop outcomes. Trail distances swept: 0.5σ, 1.0σ,
1.5σ, 2.0σ (in σ_pts units). Conservative OHLC simulation used throughout: adverse
side (stop) tested before peak update within each bar — same convention as all other
bar-scan functions.

Bar cache (logs/trailing_bar_cache.json) avoids re-fetching on reruns. 429 rate-limit
handling added: 1.0s pause between requests, retry with 10s×attempt backoff.

RESULTS BY STRATEGY:

  CSR (34 trades):
    Best trail: 0.5σ  →  +146 pts total vs fixed -593 pts  (Δ+739 pts)
    The CSR momentum signal is a spike — price moves sharply, then fades quickly.
    A tight trail locks in the initial spike gain rather than waiting for the fixed stop.

  ORB (small sample):
    Fixed stop best. ORB trades are wide-range breakouts; the move needs room.
    A trailing stop tightened too early exits breakouts that would have continued.

  VWASLR (16 trades, limited sample):
    Fixed stop marginally better in the small sample. One bad day dominated results.
    Sample too small to be conclusive; 1.5σ chosen conservatively (wider than CSR).

TRAILING STOP IMPLEMENTATION
  CSR:    0.5σ software trail + 2σ API bracket safety net
  VWASLR: 1.5σ software trail + 2σ API bracket safety net
  ORB:    no trailing stop (fixed bracket only)

  The API bracket (2σ) remains on the exchange at all times as a hard safety net for
  connectivity loss. The software trail (checked every 30s poll cycle) fires first
  under normal conditions. When trail fires: cancel brackets → close_position →
  log "TRAIL STOP" outcome.

  ActiveTrade and ActiveVwasrlTrade dataclasses both carry trail_peak and
  trail_stop_level fields updated each poll cycle. signal_monitor.py shows the
  live trail stop level and peak price in the signal/VWASLR panels.


OUTCOME CLASSIFICATION FIX (2026-03-31)
-----------------------------------------
A systematic bug was found in all bar-scan outcome classifiers: target was checked
before stop within each bar (optimistic ordering). In a volatile bar that touches
both target and stop in the same candle, this incorrectly recorded TARGET when the
stop actually fired first.

Evidence: multiple VWASLR trades in logs/vwaslr_trades.csv showed pnl_sigma = exactly
3.0 (the theoretical target multiple) despite the practice account showing a loss. This
pattern indicates _get_exit_price() returned None and the bar-scan fallback was used,
returning trade.target_price() from the optimistic check.

FIX: All four bar-scan functions updated to check stop first (conservative/adverse-first
OHLC ordering):
  trading_bot.py:    _classify_vwaslr_outcome, _classify_orb_outcome
  signal_monitor.py: _check_resolution (CSR), _check_orb_resolution

This is now consistent with backtest_trailing.py and _check_csr_trail_resolution, which
already used adverse-first ordering.

The primary path (_get_exit_price → direction-aware check) was already correct from an
earlier fix; this closes the fallback path.
